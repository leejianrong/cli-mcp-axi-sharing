Three things you pay for, every task

  • Input tokens: everything the model must read, the tool definitions and every result
  • Turns: round-trips, and each one re-sends the whole context
  • Failures: retries, or a confident wrong answer
Tokens are charged per turn. A bloated interface pays that tax every turn, not once.

The trade

  • Cheap. Nothing gets described up front; the agent just runs it
  • Universal, composable, works offline
  • Output is human-shaped, so the agent wades through verbose JSON
  • Acting and observing are separate commands, which means extra turns
  • Brittle errors, and the agent guesses at flags

AXI is design principles, not a new protocol.

Agent eXperience Interface: agent-native CLI tools that treat token budget as a first-class constraint.

The goal is the reliability of MCP at the cost profile of a CLI. It's ten principles plus a couple of reference implementations, not a runtime you install.

The headline numbers are author-published, so treat them as a strong hypothesis. We verify the findings live in a minute.

Ten principles, three groups

Efficiency

P1Token-efficient output: TOON, ~40% smaller than JSON P2Minimal default schemas: 3–4 fields, not 10+ P3Content truncation with a --full escape hatch

Robustness

P4Pre-computed aggregates: counts up front, no round-trip P5Definitive empty states, not silence P6Structured errors, exit codes, no interactive prompts

Discoverability

P7Ambient context via hooks and skills P8Content-first: no args shows live data, not help P9Contextual next-step suggestions P10Consistent --help everywhere

TOON is a compact, tabular text encoding: the density of CSV with the structure of JSON. The four highlighted principles are the ones we run on a real app next.

The four select principles, in code

P1 token-efficient output (TOON, ~40% smaller)

console.log(JSON.stringify(runs, null, 2))
console.log(renderOutput(output))

P2 minimal default schema (4 fields the task needs)

return runs   // 10+ fields per object
runs.map(r => ({ id, status, branch, logs }))

P4 pre-computed aggregate (no extra round-trip)

// (nothing; the agent burns a turn counting)
summary: summarize(loadRuns())  // "8 runs · 3 failed …"

P3 content truncation (size hint + escape hatch)

logs: r.logs
logs: full ? r.logs : truncate(r.logs, 120)

When do you use what?

CLI
when the tool already exists, usage is occasional, or you're composing shell pipelines, and token cost isn't your bottleneck.
MCP
when you need a small, stable set of typed tools inside a framework that speaks MCP natively, tool count is low, and discoverability beats token budget.
AXI
when an agent hits the tool frequently, the surface is broad, or token, cost, and latency are real constraints, and you control the tool enough to shape its output.

AXI isn't the winner, it's a set of habits. TOON output, minimal schemas, and aggregates retrofit onto any CLI or MCP server, and most of them make an MCP server better too.

Takeaways

  • The interface, not the model, often dominates agent cost. Treat token budget as a design constraint.
  • Structure and thrift aren't opposites. You can have both, and that's AXI's whole claim.
  • You can start Monday. Trim one tool's output to the fields agents actually use, and watch the turns drop.

Questions?

Aren't the benchmarks self-published?
Yes, and that's why we verified the mechanism live. The direction is robust; treat the exact figures as indicative.
Does TOON hurt readability for humans?
It's for the agent. Keep a --full or --json escape hatch for humans and tooling.
Can I get AXI's benefits without leaving MCP?
Mostly yes. Minimal schemas, aggregates, and truncation all apply to MCP tool results.
What's the catch?
You have to own and shape the tool. You can't AXI-ify a third-party API you don't control.
