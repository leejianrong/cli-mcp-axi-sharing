Three things you pay for, every task

  • Input tokens — everything the model must read: tool definitions + results
  • Turns — round-trips; each one re-sends the whole context
  • Failures — retries, or plain wrong answers
Tokens are charged per turn — a bloated interface is taxed repeatedly, not once.

Hold three numbers — tokens, turns, success. They’re our scorecard for all three approaches.

The trade

  • Cheap — no schema tax; the tool isn’t “described” up front
  • Universal, composable, works offline
  • Output is human-shaped — agent parses verbose prose/JSON
  • Action and observation split — extra turns to see what happened
  • Brittle errors; poor discoverability (agent guesses flags)
cheapbut ~86% success

It reads like a trade-off — pick cheap or reliable. The premise of AXI: that’s a false choice.

AXI is design principles, not a new protocol.

Agent eXperience Interface — agent-native CLI tools that treat token budget as a first-class constraint.

The goal: the reliability of MCP (structured, discoverable) at the cost profile of a CLI. It’s 10 principles + reference implementations — not a runtime you install.

Honesty note: it’s a young, opinionated project and the headline benchmarks are published by its author. Treat the principles as solid and the exact numbers as a strong hypothesis — we verify the direction ourselves, live.

Ten principles, three groups

Efficiency

P1Token-efficient output — TOON, ~40% smaller than JSON P2Minimal default schemas — 3–4 fields, not 10+ P3Content truncation with a --full escape hatch

Robustness

P4Pre-computed aggregates — counts up front, no round-trip P5Definitive empty states — “0 results”, not silence P6Structured errors, exit codes, no interactive prompts

Discoverability

P7Ambient context via hooks / skills P8Content-first — no args → live data, not help P9Contextual next-step suggestions P10Consistent --help everywhere

The four outlined chips are the marquee ones — the efficiency cluster plus aggregates. We’ll see them in code next, then in action on a real app. TOON = a compact, tabular text encoding: CSV’s density with JSON’s structure.

The four marquee principles, in code

P1 — token-efficient output (TOON, ~40% smaller)

console.log(JSON.stringify(runs, null, 2))
console.log(renderOutput(output))

P2 — minimal default schema (4 fields the task needs)

return runs   // 10+ fields per object
runs.map(r => ({ id, status, branch, logs }))

P4 — pre-computed aggregate (no extra round-trip)

// (nothing — agent burns a turn counting)
summary: summarize(loadRuns())  // "8 runs · 3 failed …"

P3 — content truncation (size hint + escape hatch)

logs: r.logs
logs: full ? r.logs : truncate(r.logs, 120)

Four small edits, one changed line each — lifted verbatim from the shipped src/axi.ts. Watch the output shrink in the demo.

So — when do you use what?

CLI
when the tool already exists, usage is occasional, or you’re composing shell pipelines — and token cost isn’t your bottleneck.
MCP
when you need a small, stable set of typed tools inside a framework that speaks MCP natively; tool count is low; discoverability beats token budget.
AXI
when an agent hits the tool frequently, the surface is broad, or token/cost/latency is a real constraint — and you control the tool enough to shape its output.

AXI isn’t “the winner” — it’s a set of habits. TOON output, minimal schemas, and aggregates are retrofittable to any CLI or MCP server. Most of them make your MCP server better too.

Takeaways

  • One — the interface, not the model, often dominates agent cost. Treat token budget as a design constraint.
  • Two — structure and thrift aren’t opposites. You can be both — that’s AXI’s whole claim.
  • Three — you can start Monday: trim one tool’s output to the fields agents actually use, and watch turns drop.

Thanks · Q & A

Questions?

Aren’t the benchmarks self-published?
Yes — that’s why we verified the mechanism live. Direction is robust; treat exact figures as indicative.
Does TOON hurt readability for humans?
It’s for the agent — keep a --full / --json escape hatch for humans and tooling.
Can I get AXI’s benefits without leaving MCP?
Mostly yes — minimal schemas, aggregates, and truncation all apply to MCP tool results.
What’s the catch?
You have to own and shape the tool. You can’t AXI-ify a third-party API you don’t control.
